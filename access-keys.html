Security

Access keys

This page is for Amazon Web Services (AWS) Identity and Access Management (IAM) Home Office cloud users. All cloud users in the Home Office are IAM users as their account is managed by the Public Cloud Team (PCT).

This content explains:

  • what access keys are
  • how to rotate your access keys
  • The Home Office access key policy
  • access keys for AWS services
  • temporary access keys
Back to side navigation

What are access keys?

AWS access keys are long-term credentials for IAM users. Access keys consist of an ID and a secret and provide programmatic access. Programmatic access is for services access, for example the S3 Bucket. Programmatic access is used to automate processes and actions to allow services to interact with Amazon. Access keys can be used to sign programmatic requests to the AWS Command Line Interface (CLI) or AWS Application Programming Interface (API).

Rotating access keys

Follow the AWS step-by-step guidance for rotating access keys. If you need to make any changes to your account, you can view guidance on how to raise a request with the Public Cloud Team.

You should rotate your access keys because:

  • Access keys older than 90 days are a security risk to the Home Office. The Public Cloud Team (PCT) are carrying out daily audits to find users that have access keys older than 90 days.
  • AWS set rotating access keys in their top 10 security items.
  • The CIS AWS Foundation Benchmark states you must rotate access keys to ensure that data cannot be accessed with an old key which might have been lost, cracked, or stolen. One of the main risks is it provides a window of opportunity for someone to use the keys. Rotating the keys reduces the window of opportunity smaller and stops ongoing access.

Home Office access key policy

  • The PCT are implementing the following policy for access keys, to reduce the risks to the Home Office. We will not force business areas to follow the policy. The policy allows us to highlight where there are potential risks in the organisation.
  • Only create AWS access keys where there is a clear need for programmatic access.
  • Use temporary access keys when possible.
  • Rotate access keys at least every 90 days.
  • Never share an access key with people that do not need it.
  • Store access keys securely and approved as part of a service's security assurance.
  • Do not embed access keys directly into code or repositories.
  • Tools provided by the AWS SDK and command line tools (for example, AWS credentials file) should be used instead. For more information, see the AWS documentation for using temporary credentials.
  • If an AWS access key is published to the internet (for example, GitHub), shared or lost you should follow the relevant incident management process. You should notify the public.cloud@homeoffice.gov.uk immediately. Make the affected access key inactive and delete it.
  • You should limit permissions for individual users, services or applications using AWS access keys. You should base access on least privilege and regularly review access. Administrative permissions should be given as a last resort.
  • Any IAM user that has programmatic access must enable Multi Factor Authentication (MFA).

How to use access keys in different AWS services

Amazon EC2

Where an application or AWS CLI script is running on an EC2 instance do not use access keys directly in the application. Instead define an IAM role that has the appropriate permissions for your application and attach the role to the EC2 instance. For applications using the AWS SDKs and CLI temporary, credentials are retrieved from the role automatically. For more information, see the IAM roles for EC2 documentation.

AWS Lambda

Each Lambda function has an execution role created in IAM that provides the function with the necessary permissions to run. When using the AWS SDKs, the temporary credentials are automatically retrieved from the execution role. Do not use long-lived AWS access keys when using Lambda, including providing them using the environmental variables. For more information, see the AWS Lambda execution role documentation.

IAM Users

Most IAM users do not need access keys that provide them with high levels of permissions, or multiple access keys to access multiple AWS accounts. Following best practice, IAM users should instead be provided with a single AWS access key that provides a minimal level of access. This reduces the risk if credentials are exposed.

Temporary access keys

AWS recommends the use of temporary access keys through the use of the Security Token Service (STS) instead of using long term access keys. The STS can create and provide trusted users with temporary security credentials that can control access to your AWS resources.

This is a new service and the Home Office has access keys embedded into services. This is fine as long as they are being rotated regularly. STS should be used where possible as they can be configured to last for anywhere from a few minutes to several hours. This is more secure because if credentials are stolen then they automatically expire after a short time.

For more information on STS view AWS temporary security credentials and revoking IAM role temporary security credentials.
