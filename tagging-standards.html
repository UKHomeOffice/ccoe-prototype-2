  • what a tag is

  • Home Office mandatory tags
  • Terraform example

What is a tag?

A tag is a label that you assign to an AWS resource. Each tag consists of a key and value, both of which you define for each resource, and each tag key can have only one value. You can use tags to organize your resources, and cost allocation tags to track your AWS costs on a detailed level. Tags enable you to categorise your AWS resources in different ways. For example, by purpose, owner, environment or other criteria.

Tag keys are case sensitive. Some examples are CostCenter, Environment, or Project. Tag values are also case sensitive. Some examples are 111122223333 or Production.

You can add, change, or remove tags one resource at a time from each resource’s service console, service API, or the AWS CLI. Tags identify who owns the resource and the environment where it is used. They also provide additional information and context about that specific resource.

AWS best practices

It's always best to incorporate AWS best practice when creating a tagging strategy. Some examples are:

  • do not store personally identifiable information (PII) or other confidential or sensitive information in tags
  • use a standardised, case-sensitive format for tags, and apply it consistently across all resource types
  • consider tag guidelines that support multiple purposes, like managing resource access control, cost tracking, automation, and organization
  • use too many tags rather than too few tags

Why do we want a tagging standard?

The Public Cloud Team (PCT) want to be able to track each resource back to the owner/department or project. Having a set of mandatory tags on every resource (case sensitive) makes it uniform, professional and searchable. It will also help when we need to search through logs to identify a resource.

Current problems:

  • there is no governance on what tags are used because each department has control. This makes it difficult to trace
  • it is difficult for billing to accurately charge back to each department for their usage
  • it is difficult to provide support. For example, if we identify an underutilised EC2 instance without clear tags, it makes it difficult to find the department responsible
  • the Home Office have too many tag keys and there are different variations of the same tag e.g. there are 25 tags of 'Cost Centre'

Home Office Mandatory tags

We have contacted product owners of Amazon Web Services (AWS) and Microsoft Azure to find out what tags they have used on their current resources. We have compiled a list of mandatory tags to implement across the whole estate. It's important to note that AWS have limited the number of tags per resource to 50 tags, while Azure have limited to 15. We were mindful of this and taken into account future expansion of Azure support.

Tag key

COST_CENTRE

This tag is used to allocate costs to.

  • Tag formatting - "0000000" 8 numeric characters
  • Example tag values - 01815042

ENV

Environment name standards e.g. "prod", "dev".

  • Tag formatting - lowercase alphanumerical
  • Example tag values - "prod" or "dev" or "test" or "sit" or "qat" or "sandbox"

ID

  • Tag formatting - numerical
  • Example tag values - "01"

NAME

This is a programmatic tag joining up 4 user defined tags in order to produce a unique tag reference.

  • Tag formatting - programmatic "(PROJECTID_ENV_SERVICENAME_ID)"
  • Example tag values - "publiccloud_prod_backupdb_01"

CREATED_BY

This is a Poise ID or digital username.

  • Tag formatting - lowercase alphanumeric (Poise ID only)
  • Example tag values - "mussas1"

PROJECT_ID

  • Tag formatting - lowercase alphanumeric (Poise ID only)
  • Example tag values - "public_cloud" "service_now"

SERVICE_NAME

This Normally the service tag is being applied on a resource, whether its Backup Service or a Bastion Host.

  • Tag formatting - lowercase alphanumeric
  • Example tag values "backupdb" "proxy1"

REGION

We should only be building resources in eu-west-2.

  • Tag formatting - lowercase alphanumeric (Poise ID only)
  • Example tag values - "eu-west-2"

Terraform example

The Terraform tags module can be included as follows:

 module "tags" {
                environment = var.environment
                project_service = var.project_service
                source = "git::ssh://git-codecommit.eu-west-2.amazonaws.com/v1/repos/terraform-module-aws-tags"
                }

NOTE - rather than expose SSH keys within code make use of ssh-agent to securely store the key for use with Terraform.

Follow the instructions listed under Step 3: Configure Credentials on Linux, macOs, or Unix. Then add the new identity to ssh-agent as follows: 

 $ eval $ (ssh-agent -s) ssh-add codecommit-rsa

Within your code you can add further custom tags:

 resource "aws_ecr_repository" "ecr" {
                for_each = local.repositories
                name = each.key
                image_tag_mutability = "MUTABLE" 

                image_scanning_configuration  {
                  scan_on_push = var.scan_images_on_push
                }

                tags = merge  (
                  map  (
                    "TYPE", var.type,
                    "SECONDTAG", var.secondtag
                  ),
                  module.tags.tags
                )
              }

In the code block above the mandatory tags from the module are merged with a custom tag, TYPE, detailing the type of service being provisioned.
